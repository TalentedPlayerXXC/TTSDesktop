绘声 · 前端客户端架构

Electron + React + Vite 桌面应用

语音合成客户端 · 双引擎驱动 · macOS 原生体验

基于 Electron 构建的本地 TTS 桌面客户端，通过 Vite + React 19 提供现代化 UI，MLX 后端驱动语音合成。

🏗️ 整体架构总览

绘声采用经典的 Electron 双进程模型：主进程（Main Process）负责系统集成和子进程管理，渲染进程（Renderer Process）负责用户界面。两者通过 contextBridge + ipcRenderer 安全通信，隔离级别为 contextIsolation: true / nodeIntegration: false。

🖥️ Layer 1 — Electron 主进程 (main.js)

主进程是整个应用的"操作系统胶水层"，负责后端生命期管理、IPC 接口、系统集成。

后端子进程管理

启动时通过 child_process.spawn 启动 tts_serve_mlx 二进制：

// 不传 TTS_SERVE_PORT，让后端自动找空闲端口（8000-8050）
serverProcess = spawn(serverExe, [], {
  cwd: serverCwd,
  env: {
    ...process.env,
    TTS_SERVE_HOST: '127.0.0.1',
    TTS_SERVE_LOG_LEVEL: 'warning',
    TTS_SERVE_MODELS_DIR: modelsDir,
  },
  stdio: ['ignore', 'pipe', 'pipe'],
})

端口发现机制：监听后端 stdout，解析 TTS_SERVER_PORT=xxxxx 行获取动态端口。随后轮询 /health 端点（500ms 间隔，120s 超时）确认服务就绪。

进程清理 — "诛九族"策略

应用关闭时递归杀整个进程树，确保无孤儿进程残留：

function killProcessTree(pid, signal = 'SIGKILL') {
  try {
    const out = execSync(`pgrep -P ${pid}`, { timeout: 2000 }).trim()
    if (out) out.split('\n').filter(Boolean).forEach(
      c => killProcessTree(parseInt(c), signal)
    )
  } catch (e) { /* 已死 */ }
  try { process.kill(pid, signal) } catch { }
}

另外还加了一行 execSync('pkill -f tts_serve_mlx') 作为兜底。

IPC Handlers

主进程通过 ipcMain.handle 暴露 15+ 个 RPC 接口供渲染进程调用：

  • 系统集成： select-audio（原生文件选择对话框）
  • 角色数据： get-characters-local（读取本地角色 JSON）、mongo-get-characters（MongoDB 查询）
  • 角色音频： get-character-path、get-character-emotions、get-character-preview-audio
  • 模型管理： get-server-port、get-startup-model、start-model-download、get-download-status
  • 自定义角色： migrate-custom-speaker、delete-custom-speaker、recover-custom-speakers
  • 存储/系统： get-storage-paths、get-system-info、delete-model-files（三重安全校验）
MongoDB Worker

通过 child_process.fork 启动 mongo-worker.js 子进程处理数据库查询。Worker 通过 IPC message 通信，使用 replyId 机制实现异步 RPC（15s 超时自动超时）。

模型生命周期

应用启动时检查模型文件是否存在，存在则自动加载；不存在则跳过，交给前端 ModelDownload 组件处理。打包后隐藏菜单（Menu.setApplicationMenu(null)），开发模式打开 DevTools。

🔌 Layer 2 — Preload 层 (preload.js)

通过 contextBridge.exposeInMainWorld('electronAPI', { ... }) 安全暴露能力给渲染进程：

contextBridge.exposeInMainWorld('electronAPI', {
  // 文件系统
  selectAudio: () => ipcRenderer.invoke('select-audio'),
  getCharactersLocal: () => ipcRenderer.invoke('get-characters-local'),
  getServerPort: () => ipcRenderer.invoke('get-server-port'),

  // 角色音频
  getCharacterPreviewAudio: (params) => ipcRenderer.invoke('get-character-preview-audio', params),
  getCharacterEmotions: (params) => ipcRenderer.invoke('get-character-emotions', params),
  getCharacterPath: (params) => ipcRenderer.invoke('get-character-path', params),

  // 自定义配音员
  migrateCustomSpeaker: (params) => ipcRenderer.invoke('migrate-custom-speaker', params),
  deleteCustomSpeaker: (params) => ipcRenderer.invoke('delete-custom-speaker', params),
  recoverCustomSpeakers: () => ipcRenderer.invoke('recover-custom-speakers'),

  // MongoDB
  mongo: {
    connect: () => ipcRenderer.invoke('mongo-connect'),
    getCharacters: () => ipcRenderer.invoke('mongo-get-characters'),
    getTags: () => ipcRenderer.invoke('mongo-get-tags'),
  },

  // 模型下载
  startModelDownload: (modelKey) => ipcRenderer.invoke('start-model-download', modelKey),
  getDownloadStatus: (modelKey) => ipcRenderer.invoke('get-download-status', modelKey),
  getStartupModel: () => ipcRenderer.invoke('get-startup-model'),
  getStoragePaths: () => ipcRenderer.invoke('get-storage-paths'),
  getSystemInfo: () => ipcRenderer.invoke('get-system-info'),
  deleteModelFiles: () => ipcRenderer.invoke('delete-model-files'),
  quitApp: () => ipcRenderer.invoke('quit-app'),
})

安全要点：

  • contextIsolation: true — 渲染进程无法直接访问 Node.js API
  • nodeIntegration: false — 禁止 require
  • 只暴露经过白名单的 IPC 通道，不做任何通用转发
⚛️ Layer 3 — React 渲染进程
路由设计

基于 react-router v7，5 个核心页面 + 1 个登录页：

/ → TTSComponent （默认：单人/多人/情感配音） /tts → TTSComponent （同上，独立入口） /tts-beta → TTSComponentBeta （一句话语音克隆） /voice-design → VoiceDesign （VoxCPM2 声音设计） /sound-workshop → SoundWorkshop （后期声效处理） /settings → SettingsComponent （主题/缓存/角色管理）
组件树
ErrorBoundary └─ SettingsProvider (theme context) └─ AppContent ├─ LoginComp │ ├─ LogoCanvas (canvas 粒子动效) │ └─ LoginAvatar ├─ SidebarMenu (antd Menu) ├─ Mascot (浮动吉祥物) ├─ ModelDownload (模型下载面板) ├─ DisclaimerModal (免责声明) ├─ CyberpunkLoading (加载动画) └─ Routers (react-router) ├─ TTSComponent ├─ TTSComponentBeta ├─ VoiceDesign ├─ SoundWorkshop └─ SettingsComponent
主题系统

通过 SettingsContext 管理主题偏好，支持 light / dark / system 三种模式。使用 antd 的 ConfigProvider 切换 defaultAlgorithm / darkAlgorithm，同时将 data-theme 属性设到 documentElement 上供 CSS 变量使用。

TTSComponent — 配音核心页

三种工作模式：

  • 单人配音 — 选择角色 → 输入文本 → 合成
  • 多人配音 — 添加多条配音线，每条选不同角色和文本
  • 情感配音 VoxCPM2 — 选择情感标签 + 角色 → 带情感克隆

角色数据来源：本地 characters.json + MongoDB 数据库 + 自定义配音员（localStorage + 文件系统双重持久化）。数据通过 sessionCache 内存缓存优化，避免频繁请求。

TTSComponentBeta — 一句话语音克隆

最简化的克隆流程：上传参考音频 + 输入文本 → 一步完成。适合快速试听。调用 POST /clone 端点使用 Qwen3-TTS 模型。

VoiceDesign — VoxCPM2 声音设计

纯 instruct 驱动的零样本声音设计：输入文本 + 声音描述提示词（如"御姐声线，声线偏暖略带气声"），通过 VoxCPM2 模型生成。支持 8 个预置提示词 + 自定义，输出可一键保存为自定义配音员。

SettingsComponent — 应用设置
  • 主题切换：亮色/暗色/跟随系统
  • 缓存管理：查看后端缓存状态，一键清理/按时间清理/按大小清理
  • 模型管理：查看/删除已下载模型文件（三重安全校验）
  • 自定义配音员：查看/删除本地自定义角色
  • 存储路径：显示模型目录和用户数据目录
🔗 Layer 4 — 服务层 (services/)
双 HTTP 实现 — Axios + Fetch

考虑到供应链安全，实现了两套完全独立的 HTTP 客户端：

request.tsx ── Axios fetch_request.tsx ── 原生 Fetch

两套 API 签名完全兼容，端口通过 getServerPortValue() 动态获取（electronAPI.getServerPort()）。默认使用 Axios，Fetch 版作为备胎，在 Axios 出现供应链问题时直接切换。

API 封装层 (services/index.tsx)

对后端所有 REST 端点进行 TypeScript 类型安全的封装：

函数 端点 说明
getHealth() GET /health 健康检查 + 模型加载状态
loadModel() POST /model/load 加载模型 (tts/voxcpm2)
unloadModel() POST /model/unload 卸载指定模型
unloadAll() POST /model/unload 全部卸载
clone() POST /clone 语音克隆合成
batchClone() POST /batch-clone 批量配音（可选合并）
dialogue() POST /dialogue 多角色对话
voxClone() POST /vox/clone VoxCPM2 情感克隆
voxDesign() POST /vox/design VoxCPM2 声音设计
stt() POST /stt 语音转文本
getCacheStatus() GET /cache 缓存状态
cleanupCache() POST /cleanup 清理音频缓存
模型管理策略 — "全杀"

加载新模型前先卸载全部已有模型，确保 GPU 内存干净：

let _currentModel: 'tts' | 'voxcpm2' | null = null

async function ensureModelLoaded(model: 'tts' | 'voxcpm2'): Promise<boolean> {
  if (_currentModel === model) return true
  await unloadAll()        // ← "全杀"：先清场
  try {
    const loadRes = await loadModel({ model })
    if (loadRes.data.success) {
      _currentModel = model
      return true
    }
    return false
  } catch { return false }
}

加载完成后自动后台预热：发一次空推理让 MLX 编译计算图，避免用户第一次合成时卡顿。

原始音频返回模式

关键合成接口（clone / batchClone / dialogue / voxClone / voxDesign）支持 returnRaw 参数，设为 true 时后端直接返回 WAV 二进制流（save_file: false），前端用 res.arrayBuffer() 处理，适合需要直接操作音频数据的场景。

🎛️ Layer 5 — 声音工坊 (SoundWorkshop)

基于 Tone.js 15 的实时音频效果链编辑器，支持导入已合成的音频做后期处理。

效果链架构
Tone.Player → [效果节点链] → Tone.Destination 效果节点（可叠加多个）： ├─ Gain 音量控制 ├─ Filter 高通/低通滤波器 ├─ Delay 延迟效果 ├─ Reverb 混响 (Convolver) ├─ Distortion 失真/过载 ├─ PitchShift 变调 ├─ Tremolo 颤音 ├─ Phaser 移相 ├─ Compressor 压缩器 ├─ AutoFilter 自动滤波 └─ PanVol 声像 + 音量
链式处理流程

每次效果列表变化时重建整个效果链：

  1. disposeChain() — 断开所有效果节点
  2. buildChain() — 从 Player 开始逐个连接效果节点
  3. 最后一个节点连接到 Destination（音频输出）

支持 预设系统：保存/加载完整效果链配置（localStorage），内置多个预设如 "人声增强"、"复古电台"、"空旷大厅" 等。

原生效果桥接

nativeEffects.ts 提供与 Swift/Objective-C 原生音频处理的桥接接口，可将部分计算密集效果（如实时 EQ）卸载到原生层处理，降低 JS 线程负载。

🛡️ Layer 6 — 错误处理与反馈链路
三层错误捕获
  1. ErrorBoundary — React 组件错误边界，捕获渲染过程中抛出的异常
  2. 全局 error 事件 — 监听 window.addEventListener('error')，捕获未处理的 JS 错误和资源加载失败
  3. 全局 unhandledrejection — 监听未捕获的 Promise reject，捕获异步异常
日志缓存系统

启动时劫持 console.log，将日志存入环形缓冲区（上限 300 条），发生错误时自动附带最近 30 条日志：

;(function() {
  const buffer: string[] = []
  const origLog = console.log
  console.log = function(...args: any[]) {
    buffer.push(args.map(a => typeof a === 'object' ? JSON.stringify(a) : String(a)).join(' '))
    if (buffer.length > 300) buffer.splice(0, buffer.length - 300)
    origLog.apply(console, args)
  }
  (window as any).__consoleLogs__ = buffer
})()
路径脱敏 — sanitizePath()

上报前对所有路径进行脱敏处理，确保用户隐私不被泄漏：

  • /Users/xxx/.../a/b → /a/b
  • /home/xxx/... → 相对路径
  • C:\Users\xxx\... → 相对路径
  • http://localhost:5173/src/xxx → /src/xxx
  • file:///Users/xxx/... → 相对路径
GitHub Issues 自动上报

通过 config.ts（已 gitignore）配置 GitHub 凭据：

  • Fine-grained Token，权限仅 Issues: Write
  • Token 不存在时自动弹出浏览器打开 GitHub Issues 新建页（预填标题+正文）
  • Token 存在时通过 GitHub API 直接创建 Issue
  • 自动加 labels：bug / enhancement / feedback
反馈链路
用户操作 / 崩溃 → collectFeedback() → sanitizePath() 脱敏 → formatIssueBody() 格式化为 Markdown → submitFeedback() ├─ 有 Token → GitHub API POST /issues └─ 无 Token → 打开浏览器 Issues/new
